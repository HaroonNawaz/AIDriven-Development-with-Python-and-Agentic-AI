## Key Exam Success Strategies

### Strategy 1: Focus on APPLYING, Not Memorizing
**The TOGAF Level 2 exam is ~70% APPLYING level questions**

Wrong approach:
- "What is Business Architecture?" ← Definition only

Right approach:
- "You're defining BA for a financial services company. How would you organize business capabilities?" ← Must apply

**What to do:**
- Don't just read definitions
- Answer "how would you..." questions
- Practice explaining concepts aloud
- Work through case studies

### Strategy 2: Understand the ADM Phases Deeply
**Phases A, B, C, D, E represent 75% of exam questions**

Know for EACH phase:
1. **What problem it solves**
2. **What information goes in** (inputs)
3. **What you produce** (outputs)
4. **How to execute it** (steps)
5. **Typical risks** in that phase
6. **Security considerations**

Master this by running:
```bash
python togaf_adm_phase_tutor.py all
```

### Strategy 3: Stakeholder Management is HARD
**This topic goes beyond "identify stakeholders"**

Expect questions like:
- "How would you reconcile conflicting stakeholder requirements?"
- "What role does architecture viewpoints play?"
- "How do you use trade-off analysis?"

This requires PRACTICING with scenarios, not just reading.

### Strategy 4: Security is NOW MANDATORY
**Level 1 treated security as optional; Level 2 expects it everywhere**

For EACH architecture domain (Business, Data, App, Tech):
- Know security considerations
- Understand cross-cutting security concerns
- Able to design security architecture
- Know privacy/compliance requirements

Every phase now includes security evaluation.

### Strategy 5: Master "Why", Not "What"
**Exam questions test understanding of rationale**

Not just: "What is an Architecture Contract?"
But: "Why would you use Architecture Contracts in Phase G?"

Develop deep understanding by:
1. Understanding phase purposes first
2. Then learning techniques
3. Then understanding why technique fits purpose

### Strategy 6: Time Management is Critical
**90 minutes for 55 questions = 1.6 minutes per question**

Strategy:
- Questions 1-30: 1 min each (easy flag for later)
- Questions 31-50: 2 min each (harder, more deliberation)
- Questions 51-55: 2 min each (review what you flagged)
- Final 5 min: Quick final review

**Practice this:**
```bash
# Using skills 3 & 4, time yourself
# Mark difficult questions for final review
# Don't linger on single questions
```

## Common Exam "Gotchas" (Questions People Get Wrong)

### Gotcha #1: Confusing Business Capability with Value Stream
**Question:** "What is the difference between a business capability and a value stream?"
**Wrong Answer:** "They're the same thing"
**Right Answer:**
- **Capability:** WHAT the business does (what it can do)
- **Value Stream:** HOW the business creates value (the end-to-end flow)

### Gotcha #2: Thinking Architecture Contracts are Only Legal Documents
**Question:** "What is an Architecture Contract?"
**Wrong Answer:** "It's a legal agreement with vendors"
**Right Answer:** "It's a commitment between the architecture function and other parties about what will be delivered and implemented"

### Gotcha #3: Misunderstanding Phase Purposes
**Question:** "What is the primary purpose of Phase E?"
**Wrong Answer:** "To implement the solutions"
**Right Answer:** "To identify opportunities and create an implementation strategy (Phase E is planning, not doing)"

### Gotcha #4: Thinking ADM is Waterfall
**Question:** "Why doesn't the ADM follow a strict waterfall approach?"
**Common Wrong Concept:** "It's purely sequential"
**Right Concept:** "It's iterative - phases provide feedback loops; multiple phases can execute concurrently"

### Gotcha #5: Architecture vs. Project Scope
**Question:** "How do you scope an architecture engagement?"
**Wrong Answer:** "Same as project scope"
**Right Answer:** "Architecture scope considers BREADTH (what domains) and DEPTH (what level of detail), which is different from project scope"

### Gotcha #6: Security Requirements Timing
**Question:** "When should security requirements be identified?"
**Wrong Answer:** "In Phase D (Technology)"
**Right Answer:** "Starting in Phase A, integrated throughout all phases"

### Gotcha #7: Stakeholder "Concerns" vs. "Requirements"
**Question:** "What is the difference between stakeholder concerns and requirements?"
**Wrong Answer:** "They're the same"
**Right Answer:**
- **Concern:** What the stakeholder cares about/worries about
- **Requirement:** Specific functional or non-functional capability needed
- **Addressed by:** Views that address concerns

## How to Handle These Gotchas on Exam:

When you see a question you're unsure about:

1. **Read the question carefully** - Many wrong answers confuse similar terms
2. **Look for keywords** - "What is the PURPOSE of..." vs. "What is the OUTPUT of..."
3. **Think about CONTEXT** - Phase context matters
4. **Eliminate clearly wrong** - Usually 1-2 are obviously incorrect
5. **Choose most specific** - Vague answers are usually wrong
6. **Trust your study** - If you reviewed it 3+ times, your first instinct is usually right

**Example Question:**

```
"An enterprise wants to understand how its business capabilities
are currently performing. Which technique would be most appropriate?"

A) Business Capability Mapping with Heat Maps
B) Value Stream mapping
C) Organization Mapping
D) Business Model Canvas

Approach:
- "Currently performing" suggests measurement → Heat maps (A is strongest)
- Value Stream is HOW value flows, not WHAT capabilities exist
- Organization Mapping is structure, not performance
- Business Model Canvas is business innovation, not performance
Answer: A
```
